# AI Extraction Field Confidence Rules

## 1. Objective

Define how AIM interprets AI confidence scores for extracted fields and routes them to staging review.

AI confidence is a triage signal only. It is not engineering approval, does not validate correctness by itself, and must not be used to issue an integrity decision or report.

---

## 2. Required Control Fields

Every extracted field must include:

| Field | Required | Description |
|---|---|---|
| `source` | yes | Evidence reference containing evidence code, file name, and page/table/cell reference where applicable. |
| `confidence_score` | yes | Numeric score from 0.00 to 1.00. |
| `field_status` | yes | Initial status generated by AI/validation. |
| `validation_flags` | yes | Empty array or list of validation issues. |
| `review_required` | yes | Boolean routing flag. |

---

## 3. Confidence Bands

| Band | Score Range | Initial Field Status | Review Rule |
|---|---:|---|---|
| High | `>= 0.90` | `ai_extracted` | May enter staging as ready for review, but still requires promotion by authorized human reviewer. |
| Medium | `>= 0.75 and < 0.90` | `needs_review` | Engineer review required before promotion. |
| Low | `< 0.75` | `needs_review` | Engineer review required; show warning and source evidence side-by-side. |
| Invalid | missing / non-numeric / outside 0-1 | `invalid` | Block promotion until corrected or rejected. |

---

## 4. Automatic Review Escalation

Regardless of score, review is mandatory when any of the following flags exist:

- `REQUIRED_FIELD_MISSING`
- `ASSET_TAG_MISMATCH`
- `DUPLICATE_REPORT_NUMBER`
- `UNIT_MISMATCH`
- `SUSPICIOUS_THICKNESS_VALUE`
- `MISSING_EVIDENCE_REFERENCE`
- `INVALID_DATE`
- `INVALID_STANDARD_REFERENCE_TEXT`
- `AI_ATTEMPTED_APPROVAL_OR_DECISION`

---

## 5. Field Status Values

| Status | Set By | Meaning |
|---|---|---|
| `ai_extracted` | AI/validation | Field was extracted and passed basic validation, but is not final. |
| `needs_review` | AI/validation | Field requires Engineer review before promotion. |
| `invalid` | validation | Field failed validation and cannot be promoted. |
| `rejected_by_validation` | validation | Payload or field is rejected because it violates a hard rule. |
| `approved_by_engineer` | human reviewer only | Reviewer accepts the field for staging promotion. |
| `corrected_by_engineer` | human reviewer only | Reviewer corrected the value; requires manual override reason. |
| `rejected_by_engineer` | human reviewer only | Reviewer rejects the field. |

AI must only produce initial values: `ai_extracted`, `needs_review`, `invalid`, or `rejected_by_validation`.

---

## 6. Manual Correction Rule

When a reviewer changes any AI-extracted field:

1. AIM must create a `manual_overrides` record.
2. Reviewer must provide a reason.
3. Original AI value must remain immutable.
4. Corrected value must be linked to the same evidence or to new evidence.
5. Audit event `manual_override.created` must be created.

---

## 7. Promotion Rule

A field can be promoted from staging only when:

- `field_status` is `approved_by_engineer` or `corrected_by_engineer`;
- evidence linkage is complete;
- there are no unresolved critical validation flags;
- the user has `staging.promote` permission;
- audit log is written.

---
