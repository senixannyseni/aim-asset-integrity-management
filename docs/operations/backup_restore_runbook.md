# Backup and Restore Runbook

## Purpose

Define the minimum operational backup and restore checks for AIM RC3 release-candidate operations. This runbook is operational guidance only and does not add application behavior.

## Backup Scope

- PostgreSQL database containing final structured engineering data, metadata, review state, audit logs, workflow events, error logs, and system settings.
- Object storage evidence files and report export artifacts.
- Deployment configuration and environment manifest excluding secret values.
- Release tag/commit and migration state.

## PostgreSQL Backup

1. Confirm deployment freeze/window.
2. Record current release commit/tag.
3. Run approved PostgreSQL backup command for the target environment.
4. Store backup in approved protected location.
5. Record backup file name, timestamp, database name, database host/environment, and operator.
6. Verify backup file exists and has non-zero size.
7. Record checksum verification for backup artifact.

## PostgreSQL Restore Validation

1. Restore backup to a non-production validation database.
2. Run schema/migration state verification.
3. Confirm critical tables exist: users, roles, permissions, assets, evidence_files, evidence_links, extraction_jobs, staging_records, ndt_measurements, calculation_runs, reports, report_exports, workflow_events, workflow_tasks, notification_logs, error_logs, audit_logs.
4. Run read-only smoke queries against restored data.
5. Confirm audit log records are preserved.
6. Record restore validation result and reviewer.

## Object Storage Backup / Export Verification

1. Confirm object storage bucket and prefix scope for evidence and report artifacts.
2. Export object inventory or backup manifest.
3. Verify evidence/report artifact restore considerations, including object keys, checksums, metadata, and evidence/report linkage references.
4. Compare object storage inventory against representative `evidence_files` and `report_exports` metadata.
5. Perform checksum verification for sampled evidence/report artifacts.
6. Confirm restore target can preserve original object keys or mapped restore keys with controlled metadata update procedure.

## Evidence and Report Artifact Restore Considerations

- PostgreSQL metadata and object storage artifacts must be restored consistently.
- Evidence links must not point to missing or unauthorized artifacts.
- Report export metadata must preserve checksum and artifact status.
- Restore testing must not expose raw evidence/report contents to unauthorized users.
- Signed URLs must be regenerated by AIM after restore; do not restore old signed URLs as long-lived secrets.

## Disaster Recovery Assumptions

- Recovery Point Objective and Recovery Time Objective must be defined by the deployment owner.
- Object storage and PostgreSQL backups may have separate restore procedures and timing.
- n8n workflow state is not final engineering data; AIM remains the system of record.
- Manual reconciliation may be required after partial restore to verify workflow events, notification logs, and audit logs.

## Test Restore Checklist

- [ ] PostgreSQL backup created.
- [ ] PostgreSQL backup checksum verification completed.
- [ ] PostgreSQL restore validation completed in non-production target.
- [ ] Object storage backup/export manifest captured.
- [ ] Sample evidence artifact checksum verification completed.
- [ ] Sample report artifact checksum verification completed.
- [ ] Evidence/report metadata linkage verified after restore.
- [ ] Access control verified after restore.
- [ ] Test restore result signed by IT Admin and Operations/Hypercare Owner.

## RC3-J backup/restore validation anchors

- PostgreSQL backup
- PostgreSQL restore validation
- object storage backup/export verification
- checksum verification
- evidence/report artifact restore considerations
- disaster recovery assumptions
- test restore checklist
