## 2. Required Backlog Categories

The continuous-improvement backlog must classify carryover work into at least these categories when applicable:

- security hardening;
- deployment/environment automation;
- observability and incident response;
- backup/restore/DR maturity;
- performance, scale, and data lifecycle;
- integration readiness or external CMMS cutover;
- engineering workflow UX;
- evidence governance and auditability;
- commercial/enterprise readiness;
- technical debt and maintainability.

## 3. Prioritization Rule

Every backlog item must have an owner, impact, risk/benefit statement, target release/date, and named human prioritization decision. Continuous-improvement prioritization does not reopen the approved production baseline unless named humans explicitly initiate a new release/change-control process.

AI/n8n/service actors cannot approve continuous improvement priority.
AI/n8n/service actors cannot accept improvement backlog evidence.
AI/n8n/service actors cannot close continuous improvement gaps.
AI/n8n/service actors cannot approve enterprise-readiness carryover.

## 4. Safety Rule

Do not paste secrets, JWTs, passwords, object-storage keys, signed URLs, production credentials, webhook secrets, CMMS credentials, database connection strings with passwords, private keys, confidential client evidence, raw production incident payloads, vulnerability exploit details, or unreleased commercial terms into this backlog record.
